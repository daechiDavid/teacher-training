# 기술 명세 (Tech Spec)

## 프로젝트 개요

직무연수 이수자 영수증 발급 관리를 위한 로컬 데스크톱 애플리케이션.  
담당자가 엑셀 명단을 입력하면, `이수결과`가 `"이수"`인 대상자 필터링 → 구글 시트 대조 → 영수증 자동 발급 → 구글 드라이브 업로드 → `명단` 탭 갱신까지 자동 처리한다.
사용 편의성과 검증 가능성을 우선하여, 담당자가 각 단계의 결과를 화면에서 확인한 뒤 다음 단계로 진행하는 구조로 설계한다.

## npm/pnpm 보안 정책

npm 계열 공급망 보안 이슈를 줄이기 위해 패키지 관리는 다음 원칙을 따른다.
2025~2026년 npm 악성코드 캠페인(예: Shai-Hulud 계열)은 정상 패키지 관리자의 계정 또는 배포 파이프라인을 탈취한 뒤, 악성 `preinstall`/`postinstall` 스크립트로 개발자 PC와 CI 환경의 npm/GitHub/cloud 토큰을 훔치고 추가 패키지로 전파되는 방식이 확인되었다. 이 프로젝트는 해당 공격 경로를 기본 위협 모델에 포함한다.

- 패키지 매니저는 **pnpm으로 고정**하고, `npm install`은 사용하지 않는다.
- 저장소에는 `packageManager` 필드와 `pnpm-lock.yaml`을 커밋하여 의존성 해석 결과를 고정한다.
- `.npmrc`에서 `ignore-scripts=true`를 기본값으로 설정하여 의존성 설치 시 `preinstall`, `install`, `postinstall`, `prepare` 같은 lifecycle script가 자동 실행되지 않게 한다.
- 설치 스크립트가 반드시 필요한 패키지는 사유를 확인한 뒤, `pnpm approve-builds` 또는 `onlyBuiltDependencies` 방식으로 해당 패키지만 명시적으로 허용한다.
- `package-lock.json`, `yarn.lock`은 커밋하지 않고, `pnpm-lock.yaml`만 단일 lockfile로 사용한다.
- `.npmrc`에서 `engine-strict=true`, `strict-peer-dependencies=true`, `auto-install-peers=false`를 설정하여 런타임/peer 의존성 불일치를 조기에 발견한다.
- `pnpm-workspace.yaml`의 `minimumReleaseAge: 10080` 설정으로 배포 후 7일이 지나지 않은 신규 패키지 버전은 기본 설치하지 않는다.
- 새 의존성을 추가할 때는 패키지명, 유지보수 상태, 다운로드 출처, 설치 스크립트 필요 여부를 확인한다.
- 새 의존성은 가능한 한 직접 의존성 수를 줄이고, 기능상 필요한 최소 패키지만 추가한다.
- 설치 직후 예상하지 못한 파일 변경(`package.json`, lockfile, GitHub Actions, 설정 파일, `.claude`, `.npmrc`, shell profile 등)이 생기면 악성 설치 가능성으로 보고 중단한다.
- 의존성 설치 작업 중에는 npm publish 토큰, GitHub PAT, Google OAuth client secret, 클라우드 장기 자격증명을 환경변수에 노출하지 않는다.
- CI를 추가할 경우 설치 단계에는 최소 권한 토큰만 사용하고, 패키지 publish 권한과 Drive/Sheets 운영 권한을 같은 작업에 두지 않는다.
- 배포 전에는 `pnpm audit` 또는 동등한 취약점 점검을 실행하고, 결과를 확인한 뒤 배포한다.
- OAuth 토큰, Google API 키, 샘플 개인정보, 발급 결과 PDF는 npm 패키지나 빌드 산출물에 포함하지 않는다.
- Google OAuth Client ID/Secret은 최종 사용자가 입력하지 않는다. 관리자가 앱 배포 전에 `google-oauth.local.json`, 앱 데이터 폴더/실행 파일 폴더의 `google-oauth.json`, 또는 배포 리소스의 `google-oauth.json`으로 사전 설정한다.
- 실제 OAuth 값은 Git에 커밋하지 않는다. 저장소에는 `src-tauri/google-oauth.example.json` 템플릿만 둔다.
- 데스크톱 앱에 포함된 OAuth Client Secret은 원칙적으로 완전한 비밀값이 아니다. 번들 또는 로컬 설정 파일에서 추출될 수 있으므로, PKCE·최소 scope·테스트 사용자 제한·토큰 파일 권한 제한으로 위험을 낮춘다.
- OAuth 값을 서버에서도 숨겨야 하는 수준의 요구가 생기면, 데스크톱 앱이 직접 토큰 교환을 하지 않고 별도 토큰 브로커 서버를 둔다.
- 빌드 시 환경변수로 OAuth 값을 주입해 프론트엔드 번들에 포함하는 방식은 사용하지 않는다. 데스크톱 앱 번들에서 추출될 수 있으므로 보안 경계로 보지 않는다.
- Google 설정과 OAuth 토큰은 앱 데이터 폴더에 저장하되, macOS/Linux에서는 owner-only 권한(`700` 디렉터리, `600` 파일)으로 제한한다.
- 향후 다중 사용자 배포나 더 높은 보안 요구가 생기면 OS Keychain/Windows Credential Manager 연동으로 토큰 저장소를 교체한다.

```
   ├── Google Sheets & Drive API 직접 통신 (보안 강화)
   ├── Google OAuth 토큰 안전 보관 및 로드
   ├── PDF 생성 요청 처리 및 파일 시스템 제어
   ├── 로컬 작업 상태 로그 기록 및 중복 처리 여부 판단
   └── API Quota 관리 (Throttling 및 Batching)
```

- Google API 호출 및 OAuth 토큰 처리는 **Rust 백엔드**에서 전담하여 보안성 강화
- 프론트엔드는 사용자 UI 렌더링, 엑셀 미리보기, 진행 제어에 집중
- 대량 처리에 따른 Google API 할당량(Quota) 제한 방지를 위해 백엔드 단에 호출 주기 제한(Throttling) 및 배치 업데이트(Batch Update) 장치 마련
- Drive 업로드와 Sheet 갱신은 원자적 트랜잭션이 불가능하므로, 로컬 상태 로그를 기준으로 장애 지점을 추적하고 중복 갱신을 방지한다.

| `SheetUpdater` | 업로드 성공 후 `명단` 탭의 발급횟수·과정명·발급날짜·개별 PDF 링크 자동 기록 |
| `JobLogStore` | 대상자별 처리 상태, Drive 파일 ID, Sheet 갱신 여부, 오류 기록 |

## 구글 스프레드시트 구조

### `명단` 탭

| 컬럼 | 내용 |
|------|------|
| A | 연번 |
| B | 학교명 |
| C | 이름 |
| D | 전화번호 |
| E | 발급횟수 |
| F | 과정명1 |
| G | 발급날짜1 |
| H | 링크1 |
| I | 과정명2 |
| J | 발급날짜2 |
| K | 링크2 |

- 매칭 기준: **B열(학교명) + C열(이름) + D열(전화번호)** 동시 일치
- 발급횟수가 0이면 `과정명1`, `발급날짜1`, `링크1`에 연수과정명, 날짜, 개별 PDF 링크 기록
- 발급횟수가 1이면 `과정명2`, `발급날짜2`, `링크2`에 연수과정명, 날짜, 개별 PDF 링크 기록
- 발급횟수가 2 이상이면 자동 발급 대상에서 제외한다.
- 실제 구현은 고정 열 번호가 아니라 1행 헤더명을 기준으로 컬럼 위치를 찾는다.
- 필수 헤더가 없거나 이름이 일치하지 않으면 처리를 시작하지 않고 오류로 안내한다.

### 필수 헤더

`명단` 탭 1행에는 다음 헤더가 모두 있어야 한다.

`연번 | 학교명 | 이름 | 전화번호 | 발급횟수 | 과정명1 | 발급날짜1 | 링크1 | 과정명2 | 발급날짜2 | 링크2`

## 시트 정합성 검사

업무 처리 전에 `명단` 탭의 기존 발급 기록을 검사한다.

- 발급횟수가 0이면 `과정명1`, `발급날짜1`, `링크1`, `과정명2`, `발급날짜2`, `링크2`는 모두 비어 있어야 한다.
- 발급횟수가 1이면 `과정명1`, `발급날짜1`, `링크1`은 모두 있어야 하고, `과정명2`, `발급날짜2`, `링크2`는 비어 있어야 한다.
- 발급횟수가 2이면 `과정명1`, `발급날짜1`, `링크1`, `과정명2`, `발급날짜2`, `링크2`가 모두 있어야 한다.
- 위 조건에 맞지 않는 행은 자동 처리하지 않고 수동 확인 대상으로 표시한다.
- 발급횟수가 숫자가 아니거나 음수이면 자동 처리하지 않고 수동 확인 대상으로 표시한다.

## 정규화 및 매칭 규칙

- 이름, 학교명, 연수과정명은 앞뒤 공백을 제거한 문자열로 비교한다.
- 전화번호는 숫자만 남긴 뒤 비교한다.
- 정규화된 전화번호는 국내 휴대전화 기준 `010`으로 시작하는 11자리 값을 정상값으로 본다.
- 전화번호가 비어 있거나 11자리 정상값으로 정규화되지 않으면 자동 발급하지 않고 수동 확인 대상으로 분리한다.
- 입력 엑셀과 `명단` 탭 모두 정규화한 `이름 + 학교명 + 전화번호` 조합이 정확히 1개 행으로 식별되어야 자동 처리한다.
- 같은 이름과 학교명이 중복되는 경우 파일명은 `이름_학교명_전화번호뒤4자리.pdf`로 저장하고, 그렇지 않으면 `이름_학교명.pdf`로 저장한다.
- Drive 연수별 폴더명은 `연수과정명_발급날짜` 형식으로 생성한다.
- 파일명과 폴더명에 사용할 수 없는 문자는 공백 또는 안전한 문자로 치환한다.

## 재발급 금지 규칙

- 재발급 금지 기준 키는 `연수과정명 + 정규화이름 + 정규화학교명 + 정규화전화번호`이다.
- `명단` 탭의 `과정명1` 또는 `과정명2`에 같은 연수과정명이 이미 기록되어 있으면 자동 재발급하지 않는다.
- 로컬 상태 로그에 같은 키가 `updated` 상태로 존재하면 추가 안전장치로 자동 재발급하지 않는다.
- `명단` 탭의 발급횟수가 2 이상이면 자동 재발급하지 않는다.
- 발급횟수가 0 또는 1이어도 같은 연수과정명과 같은 대상자가 이미 완료 처리된 기록이 있으면 자동 재발급하지 않는다.
- 재발급이 필요한 예외 상황은 자동 처리하지 않고 담당자가 수동으로 기존 기록을 확인한 뒤 별도 처리한다.

## 로컬 작업 상태 로그

중단 복구와 중복 발급 방지를 위해 대상자별 처리 상태를 로컬에 기록한다.

| 필드 | 내용 |
|------|------|
| `run_id` | 한 번의 실행을 식별하는 ID |
| `training_id` | 연수 또는 회차 식별자 |
| `person_key` | 이름+학교명+전화번호 기반 대상자 키 |
| `sheet_row` | Google Sheet 행 번호 |
| `receipt_path` | 생성된 로컬 PDF 경로 |
| `drive_file_id` | 업로드된 Drive 파일 ID |
| `drive_file_link` | 업로드된 개별 PDF 공유 링크 |
| `sheet_updated` | Sheet 갱신 완료 여부 |
| `status` | pending / generated / uploaded / updated / failed |
| `error_message` | 실패 시 오류 내용 |
| `updated_at` | 마지막 상태 변경 시각 |

발급 로그를 별도 시트로 생성하는 것은 필수 워크플로우에는 포함하지 않는다. `명단` 탭의 `과정명/발급날짜/링크` 기록만으로 재발급 방지와 개별 추적이 가능하므로, 별도 로그 시트는 추후 감사 이력이나 상세 장애 분석이 필요할 때 확장한다.

## 빌드 및 배포

- **개발 환경**: macOS
- **타겟 플랫폼**: macOS 우선, Windows는 후순위 검토
- **배포 형식**: `.dmg` (macOS), Windows 확정 시 `.msi` 또는 `.exe`
- 앱 사용자는 Google 로그인으로 본인 계정의 Sheets/Drive 접근 권한을 승인한다.
- OAuth 토큰은 앱 데이터 폴더에 저장하고 macOS/Linux에서는 owner-only 파일 권한으로 제한한다. 향후 보안 요구가 높아지면 OS Keychain/Windows Credential Manager로 교체한다.
- 업로드 대상은 0단계 기본 설정에서 저장한 Google Drive 상위 폴더 아래 생성되는 `연수과정명_발급날짜` 폴더이다.
- 업로드된 개별 PDF 파일은 `링크가 있는 사용자: 뷰어` 권한으로 공유한다.
- 처리 결과 화면에는 업로드된 PDF가 링크 뷰어 권한으로 공유되었다는 안내 문구를 표시한다.
